Introduction

This HTML file is a template you can edit in an app like Notepad, or BBEdit, or something more complicated. This template is free to use with or without credit (though credit to me under the name 'Speak the Sky', with or without a link to where the template can be found online, is appreciated). If you use, or even build on this template, please drop a link to your work in the comments on the itch page!

To use this template to build a game file, you will need to write HTML (for the text's structure) and a little bit of CSS (for the styling). Writing HTML and CSS is not that hard, actually, even if it's unfamiliar and maybe a little daunting at first. In fact, it's pretty easy once you get the hang of it, especially if you're doing straightforward static pages like this one. Anyway, I'd recommend the following 3 resources for beginners:

  • the W3C tutorials, a beginner-friendly set of tutorials and references for how HTML works, what kinds of tags there are, how to write a document, etc., plus tutorials on CSS (for styling your file) and JavaScript (for adding/editing functionality like the dice roller, card deck, and menu tabs).
  • the MDN web docs, a little bit more general, technical, and up-to-date (sometimes) than the W3C tutorials, plus it has examples in-line.
  • search engines, where you can sift through irrelevant bullshit on Stack Exchange or wherever and probably find an answer for any question you have (just pay attention to when the answers were written, since they may've become obsolete).

You might think "why bother?" and the reason you might wanna bother is that properly-written HTML supported with CSS and basic JS is highly flexible, accessible, customisable, and functional, and can provide experiences you can't get with other formats.

Caveat 1: I haven't properly edited this or anything. The text in this file is a very rough guide to the underlying code and not any kinda tutorial on how to write HTML, CSS, or JS.

Caveat 2: This file doesn't have every possible feature it could have, or even every feature that'd be nice to have while still keeping it relatively lo-fi (e.g. sticky section headers or breadcrumbs that help identify where you are at any point in the text). If you wanna build on it by adding or refining features then please share your work!

Caveat 3: You can't write any kind of HTML in this template and expect it to look flawless with a 100% guarantee. Look at how the underlying HTML's written in this file to see how I wrote it, and what the CSS is written to style. That said, the stuff you'll need to use for your game text is pretty straightforward, mostly <h2>, <h3>, <section>, <p>, and <hr/> elements and some text-formatting elements like <strong>, <em>, and <span>s with classes.

Disclaimer: I won't help you with general coding questions. I'm not tech support for your browser or a teacher for your programming education.

Metadata

Don't forget to edit your game file's metadata. That means the following data near the top of the <head> element near the start of the file (should be one of the first things you see when opening it):

  • title in the <title> tags
  • writer name in the content attribute of the <meta name="writer"/> tag
  • description in the content attribute of the <meta name="description"/> tag

If you don't change these then they'll either be empty (if you're using the base file) or the same as this file, which probably isn't what you want (especially considering that the <title> text will appear in the browser tab when someone opens your file).

Styles

The file comes with a range of stylistic rules, which you can find between the <style> tags in the HTML head. Some of these are pretty straightforward, like the headings, colours, and text properties. Others, like the oracle-grid styles (see examples), are little more complex. The rules are split into a few categories:

  • variables: colours, some text properties, sizes of basic structural elements like the main text column, etc. Changes made here will ripple through the whole file.
  • fonts: currently, only the font-face declaration for the custom version of Dicier, embedded in base64-encoded form.
  • basic structure: mostly for the size and position of the structural elements of the file like the cover, main content, header and footer, etc.
  • menu: for the menu controls and menu tab contents.
  • text: for headings, regular text, oracles, horizontal rules (under "section decoration"), italics, game icons, etc.
  • media: in this file, this is only for the SVG menu tab icons and logo, but you could put other media tag CSS here (for images, video, audio, etc.).
  • display: for changing the layout when the viewport is under 800px and 500px wide.
  • themes: for the light and dark themes, including a few rules for different screen sizes.
  • theme transition: for transitions between themes (however, these rules will cover all transitions, not just theme transitions).
  • reset theme side-effects: undoes some potentially less-legible combinations of text and background colour, among a couple other things.

The Menu

To the left of the text (or at the bottom on narrower screens) you'll find a menu bar with a few tabs (and the option to close all tabs). This menu is sticky, that is, it stays in the same place on the page as long as you're scrolling in its vertical space (basically, it won't appear in the game cover area at the top of the file). By default, the only thing visible is just the menu tab options (I've made SVG icons, but you could also use text). Clicking a tab's icon brings up that tab; clicking the X closes all the tabs.

In this file, the menu contains three tabs:

  • Contents
  • Settings
  • Tools

To remove a tab, you'll need to delete the following:

  • its input and label at the start of the menu div
  • the menu tab itself further down the menu div
  • the tab name from the menuSwitch function in the script section at the end of the HTML body

To add a new tab, do the opposite. Whether adding or removing tags, you'll probably also want to change the "#menu-controls label" width property in the tablet (max-width: 799px) media query as well. This basically controls how wide the menu tab buttons are in tablet and mobile view, and by default is set to 25% or one quarter, because by default there are 3 tabs and the close-tabs button.

Menu tab: Contents

The Contents tab has a list of navigation links leading to some, or maybe all major sections of your file, with sub-lists leading to sub-sections.

To set these up in your own text, you'll need to add id attributes to the HTML elements you want to link to from the contents tab. You can see how it's done by looking at this file's HTML (check out the section headings and compare them to the <nav> element containing the contents list). You can link to anything, not just heading elements, and sections or subsections of your text don't necessarily have to have these links.

Menu tab: Settings

The Settings tab has controls for changing how the file displays: colour scheme, font type, text width, and italics style. You can switch to a new setting for a given control by clicking or focusing on that setting.

This file comes with 2 colour schemes (light and dark themes, which smoothly transition over half a second), 2 font options (serif and sans-serif), 3 text widths (thin, default (about 70 characters per line), and wide), and 5 italics display options (italicised, marked with asterisks or slashes, highlighted, or unmarked).

The reason for the italics control and its many settings is that I sometimes see dyslexic people mention how italics can be much harder to read than regular text, but I figure what people would prefer instead could be any number of things.

There's also some script near the start and end of the file to store and re-apply settings when you refresh or reopen the file (the data's saved to your browser's local storage when you change a given setting), so it'll eventually reset if you don't open the file for a while).

To remove a setting, remove the fieldset for the setting and its options from the menu div, then go to the script section at the bottom of the HTML and remove the matching function (e.g. italicsSwitch for removing the italics control), event listeners, and property's default setting (this is set in the event listener near the end of the file that listens for "DOMContentLoaded", that is, your browser loading the file). You'll probably have trouble adding settings unless you have programming experience, but my game Like Skyscrapers Blotting Out The Sun, which I made this template for, has a couple extra controls (text alignment, paragraph division) and a third colour scheme (bookwove) that you can copy over if you think they're relevant to your game text.

Menu tab: Tools

The Tools tab has a simple dice roller (d6 only) and card deck (poker only, no jokers). The file will remember your last 6 rolls as well as the current state of the deck and the last card you drew.

Like I said, these tools are super-simple—they don't do anything beyond generate random numbers or shuffle an array of text and pull bits of that text off the array. No fancy graphics, no integration with game oracles or anything like that.

Responsiveness

The file is set up to display slightly differently below 800 pixels wide and below 500 pixels wide. You can test this on a desktop or laptop computer by shrinking the window, or by opening the file on other devices (e.g. phone, tablet).

Below 800px, the menu will shift to the bottom of the window, and each menu tab will switch to 2 columns instead of 1.

Below 500px, the layout will become more compact, with the body text and oracle icons shrinking in size.

A few other minor changes also happen at each breakpoint.

The 800px breakpoint styling's meant to cover mobile screens and portrait-oriented tablet screens. The 500px breakpoint styling covers mobile screens. Landscape-oriented tablets may be wide enough for the regular styling.

Disclaimer: I don't know that much about responsive design yet, so this stuff can probably be improved on a lot, but it works well enough.

Oracle Grids

Rather than use HTML tables, this file uses CSS to change description lists into table-like grids. Description lists involve 3 elements:

  • <dl>, which wrap around the description list
  • <dt>, which wrap around terms to be described
  • <dd>, which wrap around descriptions

This file adds a fourth: the horizontal rule element <hr/>, which goes between oracle entries as a spacer (I removed the border on these <hr/>s, so the only thing they provide is space, but you could easily add the border back in).

In this case, the 'terms' are actually card suits and ranks, and the 'descriptions' are actual oracle entries (or parts of oracle entries, since you can have multiple descriptions per term). This file has three different oracle grid types in it:

  • suits-then-ranks, where the suit gives you a category and the rank gives you details.
  • ranks-then-suits, where the rank gives you a category and the suit gives you details.
  • ranks-only, where the rank gives you the full oracle entry.

The first two oracle grid types have two components: the category itself in one <dd> element, then the details in a second <dd> that actually includes a small <dl> of its own. The third oracle grid type is more straightforward; each rank is a <dt> element and each entry is a <dd> element straight after. If this is getting confusing then try looking at the HTML in an editor like BBEdit. It should make more sense seeing it actually laid out.

The oracle grids have a bit of dedicated CSS that let you customise the entries, e.g. adding a 2-component entry to a rank-only oracle (like the entry a mutual revelation in the Event Oracle in Like Skyscrapers), or turning an entry in a two-component oracle into a 1-component entry (like the last entry in the Footnote Oracle in Like Skyscrapers). You can do this by adding the "one-component-entry" class to a <dt> element in a two-component oracle, or "two-component-entry" to a <dt> element in a one-component oracle. Just use the new number of <dd> elements for that entry and the oracle will lay out exactly as it should. Again, if you need an example, look at the Like Skyscrapers oracle entries mentioned above.

Lastly, all the oracles have id-links built into the main icons for each entry (e.g. the suit icons in a suits-then-ranks oracle grid). This means that if you click that icon, your url will be set to that entry. This relies on having unique ids and matching hrefs for each entry (in the <dt> element for that entry), so if you copy and paste the same oracle grid you must edit the ids and hrefs in the copied oracle grid to make the links work, because ids must be unique in a single HTML file.

The icon font (used for dice and card suits and ranks) is Dicier v1.53 Write Skyscrapers version by Speak the Sky, used under the terms of the CC BY 4.0 license.

The combined icon font can be found in base64 form in the raw HTML of this file (between the <style> tags near the top).
